# GitHub OIDC Setup

This document explains how to connect GitHub Actions to AWS using OpenID Connect (OIDC) federation — the mechanism that allows workflows to assume IAM roles without storing any AWS credentials in GitHub.

---

## How It Works

```
GitHub Actions runner
  │
  ├── Requests a short-lived OIDC token from GitHub
  │
  └── Calls AWS STS AssumeRoleWithWebIdentity
        │
        ├── AWS validates the token against the GitHub OIDC provider
        └── Returns temporary credentials (15 min – 1 hr)
```

The IAM roles restrict which GitHub repository and environment can assume them, so a workflow in `org/other-repo` cannot obtain credentials for your deployment role.

---

## Step 1 — Apply Terraform

The `github_oidc_roles` Terraform module handles provider and role creation automatically.

```bash
cd infra/environments/dev
terraform apply
```

This creates:

| Resource | Description |
|---------|-------------|
| `aws_iam_openid_connect_provider` | GitHub OIDC provider (once per account) |
| `infra-deployer` role | Can manage Terraform state and all infra resources |
| `backend-deployer` role | Can update the Lambda function code only |
| `frontend-deployer` role | Can write to the frontend S3 bucket and invalidate CloudFront |

**Note:** If you are applying both dev and prod environments in the same AWS account, set `create_oidc_provider = false` in `prod/terraform.tfvars`. The OIDC provider is account-scoped and must only be created once.

After applying, note the role ARNs:

```bash
terraform output infra_deployer_role_arn
terraform output backend_deployer_role_arn
terraform output frontend_deployer_role_arn
```

---

## Step 2 — Create GitHub Environments

In your GitHub repository:

1. Go to **Settings → Environments**.
2. Create two environments: `dev` and `prod`.
3. Optionally configure protection rules on `prod` (required reviewers, deployment branches).

---

## Step 3 — Add Secrets and Variables

For each environment (`dev` and `prod`), add the following. Values come from `terraform output`.

### Secrets

Secrets are encrypted and only exposed to workflows at runtime.

| Secret name | Value source |
|------------|-------------|
| `INFRA_ROLE_ARN_DEV` / `INFRA_ROLE_ARN_PROD` | `terraform output infra_deployer_role_arn` |
| `BACKEND_ROLE_ARN_DEV` / `BACKEND_ROLE_ARN_PROD` | `terraform output backend_deployer_role_arn` |
| `FRONTEND_ROLE_ARN_DEV` / `FRONTEND_ROLE_ARN_PROD` | `terraform output frontend_deployer_role_arn` |
| `VITE_API_URL_DEV` / `VITE_API_URL_PROD` | `terraform output api_url` |
| `VITE_COGNITO_USER_POOL_ID_DEV` / `VITE_COGNITO_USER_POOL_ID_PROD` | `terraform output cognito_user_pool_id` |
| `VITE_COGNITO_CLIENT_ID_DEV` / `VITE_COGNITO_CLIENT_ID_PROD` | `terraform output cognito_client_id` |
| `VITE_COGNITO_HOSTED_UI_DOMAIN_DEV` / `VITE_COGNITO_HOSTED_UI_DOMAIN_PROD` | `terraform output cognito_hosted_ui_domain` |
| `VITE_COGNITO_REDIRECT_URI_DEV` / `VITE_COGNITO_REDIRECT_URI_PROD` | `terraform output frontend_url` |

### Variables

Variables are plain text, visible in workflow logs (use these for non-sensitive identifiers).

| Variable name | Value source |
|--------------|-------------|
| `S3_BUCKET_NAME_DEV` / `S3_BUCKET_NAME_PROD` | `terraform output frontend_bucket_name` |
| `CLOUDFRONT_DISTRIBUTION_ID_DEV` / `CLOUDFRONT_DISTRIBUTION_ID_PROD` | `terraform output cloudfront_distribution_id` |
| `LAMBDA_FUNCTION_NAME_DEV` / `LAMBDA_FUNCTION_NAME_PROD` | `terraform output lambda_function_name` |

## Step 4 — Verify a Workflow Run

Trigger a workflow manually from the **Actions** tab to confirm credentials are working:

1. Go to **Actions → Deploy Backend → Run workflow**.
2. Select the `main` branch and `dev` environment.
3. Check that the **Configure AWS credentials** step completes without error.

If the step fails with `Not authorized to perform sts:AssumeRoleWithWebIdentity`, verify:

- The role ARN secret is correct.
- The IAM role's trust policy `Condition` matches the repository name and environment name exactly.
- The GitHub environment name in the workflow matches the environment created in Step 2.

---

## IAM Trust Policy Reference

The trust policy generated by the `github_oidc_roles` module looks like this:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Federated": "arn:aws:iam::123456789012:oidc-provider/token.actions.githubusercontent.com"
      },
      "Action": "sts:AssumeRoleWithWebIdentity",
      "Condition": {
        "StringEquals": {
          "token.actions.githubusercontent.com:aud": "sts.amazonaws.com",
          "token.actions.githubusercontent.com:sub": "repo:my-org/my-repo:environment:dev"
        }
      }
    }
  ]
}
```

The `sub` claim locks the role to a specific repository **and** GitHub environment. A workflow running against the `prod` environment cannot assume the `dev` role, and vice versa.

## Adding a New Environment

1. Apply Terraform for the new environment (e.g., `infra/environments/staging/`).
2. Create a `staging` GitHub environment in repository settings.
3. Add the role ARNs and output values as `_STAGING`-suffixed secrets/variables.
4. Update the workflow files to include `staging` in the environment matrix.

---

## Rotating or Revoking Access

Because credentials are temporary and issued per workflow run, there is nothing to rotate. To revoke access:

- **Temporarily**: disable the GitHub environment or add a required reviewer to block deployments.
- **Permanently**: delete the IAM role via Terraform (`terraform destroy -target=module.github_oidc_roles`).
